# S13.4.1 — llama.cpp vs ollama comparison

**Phase:** P13.4 (LLM backend strategy)
**Estimate:** 1 week
**Status:** stage placeholder.
**Shape:** investigation, feeds question Q21.

## Context

`ideas.md` line 118: "llama.cpp vs ollama".

ollama is a wrapper / orchestration layer over llama.cpp
(plus other backends) with: a model registry, a one-
command pull/run flow, REST API close-but-not-identical to
llama-server, model hot-swap as a first-class operation.

Question: does ollama replace acva's M8A model-controller
sidecar, or is it incompatible with pillar #1 / pillar #5
(don't write a custom HTTP wrapper around a backend that
already ships one)?

## Tasks

### T13.4.1.1 — Feature parity matrix

For each acva-relevant feature:

- streaming chat-completions API → both
- GBNF grammar (M9.P9.4) → llama.cpp yes; ollama partial
- multi-slot parallel inference (M12.P12.5) → llama.cpp
  yes; ollama unclear
- model hot-swap → ollama native; llama.cpp via M8A
  sidecar
- VRAM management → ollama auto; llama.cpp manual
- API surface stability → llama.cpp evolving; ollama
  stable

### T13.4.1.2 — Pillar fit

Pillar #1 (process isolation) — ollama runs as its own
process / container. ✓
Pillar #5 (don't write custom wrapper) — ollama IS a
wrapper. So adopting ollama means deleting M8A's model-
controller sidecar (which exists exactly because llama
.cpp doesn't expose hot-swap). ✓ for ollama.

### T13.4.1.3 — Migration cost

If we pick ollama:
- delete `packaging/model-controller/`
- rewrite `LlmClient` to ollama's chat API quirks
- rewrite registry → ollama's model name conventions
- M9.P9.4 GBNF support — partial; might need a fork

If we keep llama.cpp:
- M8A stays.
- continue manual model-controller maintenance.

### T13.4.1.4 — Recommendation

Capture in this stage's closure note. Then file Q21
resolution in `plans/open_questions.md`.
