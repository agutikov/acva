# S13.4.2 — Model-controller scope under each backend

**Phase:** P13.4 (LLM backend strategy)
**Estimate:** 3-5 days
**Status:** stage placeholder.
**Shape:** investigation.

## Context

The M8A model-controller is a Go sidecar that holds the
docker socket privilege and recreates llama-server with
new flags / model file. Its existence is justified by
"llama.cpp doesn't hot-swap models". If we adopt ollama
(see S13.4.1), the sidecar becomes dead code; if we keep
llama.cpp, the sidecar's scope might still grow (e.g.
managing slots for M12.P12.5).

## Tasks

### T13.4.2.1 — Three scope variants

- **Variant A** — keep llama.cpp, sidecar stays at v1.
- **Variant B** — keep llama.cpp, sidecar grows to manage
  slots (M12.P12.5), GPU eviction (S12.5.4), and
  potentially RVC (S9.5.3) lifecycle.
- **Variant C** — adopt ollama, delete sidecar, accept
  ollama's lifecycle model.

Document the implementation cost + maintenance cost of each.

### T13.4.2.2 — Hybrid?

Could acva talk to ollama AND keep the sidecar for
non-LLM model management (Speaches CUDA-OOM remediation,
RVC lifecycle)? Maybe. Sketch.

### T13.4.2.3 — Resolve Q21
