# S13.1.2 — Speaches as a systemd service

**Phase:** P13.1 (Linux native packages)
**Estimate:** 3-4 days
**Status:** stage placeholder.
**Shape:** packaging task.

## Context

`ideas.md` line 120: "speeches systemd service ???".

Today Speaches runs as a Compose container in the dev
default. Production-style operators may prefer running
Speaches as a systemd unit alongside acva's own systemd
units (the `acva.target` chain landed in M2 stretch / M8C).

## Tasks

### T13.1.2.1 — Unit file

`packaging/systemd/speaches.service` (per-user via
`systemd --user`). Runs the Speaches container via
`podman` (avoids docker daemon dependency); falls back
to docker if podman absent.

### T13.1.2.2 — Required envs / volumes

Mirror the Compose `environment:` / `volumes:` blocks:
GPU device, model cache mount, port binding, the
non-negotiable `WHISPER__TTL=-1`.

### T13.1.2.3 — `acva.target` integration

Add a `Wants=speaches.service` line so `systemctl --user
start acva.target` brings up Speaches too. `Before=acva
.service` so Speaches is up before acva probes.

### T13.1.2.4 — Native install path

If operator prefers no container, document running
Speaches's Python directly via systemd unit. Likely
fragile across distros; opt-in only.

### T13.1.2.5 — Acceptance

Fresh VM with the .deb installed: `systemctl --user
enable acva.target` brings up llama, Speaches, and acva
in the right order; `acva demo health` passes.
