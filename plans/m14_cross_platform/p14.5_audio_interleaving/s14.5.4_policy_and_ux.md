# S14.5.4 — Interleaving policy + UX (Q27)

**Phase:** P14.5 (audio interleaving)
**Estimate:** 3-4 days
**Status:** stage placeholder.
**Shape:** decision + config surface.

## Context

Per-OS stages (S14.5.1-3) handle the *mechanism*. This stage
unifies the *policy* — when does acva duck, by how much, and
what's the operator-visible UX?

## Tasks

### T14.5.4.1 — Q27 — policy choice

Three options:
- **A)** Always duck while assistant speaks. Simple;
  annoying when listening to music *and* asking acva
  background questions.
- **B)** Only duck if user is engaged (recently spoke).
  Smart; needs heuristic.
- **C)** Operator-configured per personality. Maximum
  flexibility; most config surface.

Default A. Decision lands in
`m14_cross_platform/open_questions.md` Q27.

### T14.5.4.2 — Config surface

```yaml
audio:
  ducking:
    enabled: true
    amount_db: 12
    attack_ms: 100
    release_ms: 200
    policy: always   # always | engaged | manual
```

### T14.5.4.3 — UX

Operator can disable mid-session via
`/duck?enabled=false` HTTP control. Useful when acva is
narrating along with music.

### T14.5.4.4 — Acceptance

End-to-end: across all three OSes (where stages have
shipped), the same config produces equivalent behavior.
