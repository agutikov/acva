# S15.1.1 — Module ABI design

**Phase:** P15.1 (module ABI)
**Estimate:** 3-4 weeks
**Status:** stage placeholder.
**Open question:** Q9 (ABI stability) — decided here.

## Likely tasks

### T15.1.1.1 — ABI surface

`include/acva/abi.h` — pure C, no C++ in the surface. Defines
module-loading entry points (`acva_module_init`,
`acva_module_query_capability`, `acva_module_shutdown`),
capability types (audio backend, memory store, TTS engine,
tool runtime), and per-capability vtables.

### T15.1.1.2 — Versioning policy (Q9)

Strict semver vs FFmpeg-style "minor breaks ABI" vs pre-1.0
unstable. Pick + document in `m15_modularity_plugin_platform/open_questions.md`.

### T15.1.1.3 — Lifecycle

Modules dlopen()'d at startup, called via vtable, dlclose()
on shutdown. Failure modes: incompatible version → log
warn + skip; init failure → log error + skip; crash →
caught at the call boundary, module disabled for the
session.

### T15.1.1.4 — Capability negotiation

A single .so can register multiple capabilities. Module
declares what it provides; orchestrator wires the first
matching provider per capability slot. Operator override
via config.

### T15.1.1.5 — Tests

Unit: ABI header compiles in pure C; vtable invariants;
version negotiation logic. Integration: a hello-world
module .so that registers a fake audio backend; load it,
verify it intercepts.
