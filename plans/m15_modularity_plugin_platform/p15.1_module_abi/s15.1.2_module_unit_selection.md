## Tasks

### T15.1.2.1 — Subsystem inventory

For every subsystem under `src/`, document:
- size (LOC)
- coupling (events in/out, shared state)
- swap-likelihood (would an operator ever want a different
  impl?)
- ABI complexity (could this fit a C ABI?)

Subsystems: audio (capture/resample/APM/VAD/endpointer/
loopback), memory (Repository/Database/MemoryThread),
LLM (LlmClient/PromptBuilder), STT (Speaches client +
realtime), TTS (Bridge + clients), playback, dialogue
(Manager + FSM + SessionManager + BargeInDetector),
orchestrator helpers, observability, supervisor.

### T15.1.2.2 — Per-subsystem yes/no

Build a matrix:

| Subsystem | Module-able? | Worth it? | ABI cost |
|-----------|--------------|-----------|----------|
| audio backend | yes | yes | low |
| memory store | yes | yes | medium |
| TTS engine | yes | yes | low |
| STT engine | yes | yes | low |
| LLM client | yes | yes | low |
| AEC (APM) | yes | maybe | medium |
| FSM | technically yes | NO — too coupled | high |
| PromptBuilder | yes | maybe | low |
| Manager | NO | NO | very high |
| Endpointer | yes | maybe | low |
| Address detector | yes | YES | low |

Justify each call.

### T15.1.2.3 — Resolve Q26

Final module list lands in
`m15_modularity_plugin_platform/open_questions.md` Q26.

### T15.1.2.4 — Pillar callout

`ideas.md` line 62: "do not mixup modularity and
configurability". Document the discipline: a module
boundary should *only* exist for things operators want to
swap with a *different binary*. Configurable behavior
(personality, voice, prompts) does not need a module
boundary.
