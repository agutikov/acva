# S15.3.1 — Plugin registry + ops

**Phase:** P15.3 (plugin registry + ops)
**Estimate:** 2-3 weeks
**Status:** stage placeholder.

## Tasks

### T15.3.1.1 — Discovery

`~/.local/share/acva/plugins/*.so` auto-discovered at
startup. Manifest sidecar (`<name>.toml`) declares
capabilities, version, source URL, signature.

### T15.3.1.2 — `acva plugins` CLI

`list` (installed + enabled state), `enable <name>`,
`disable <name>`, `info <name>`, `verify` (sig check).

### T15.3.1.3 — Permissions / sandbox

Plugins are native code — they can do anything. M11.P11.1
Q6 sandboxing decision applies if plugins host third-party
code. Most plugins ship from operator's own build, so
sandboxing is opt-in.

### T15.3.1.4 — M10 Web UI surface

Plugin install / enable from the Web UI. Permissions UX
similar to tools (M11.P11.5).

### T15.3.1.5 — Acceptance

Operator builds a third-party module .so, drops in
plugins/, runs `acva plugins enable foo`, restarts, sees
the module load + take effect. End-to-end without
recompiling acva.
