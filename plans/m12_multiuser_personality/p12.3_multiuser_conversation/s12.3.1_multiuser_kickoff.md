# S12.3.1 — Multi-user conversation handling

**Phase:** P12.3 (multi-user conversation)
**Estimate:** 3-4 weeks
**Status:** stage placeholder.

## Likely tasks

### T12.3.1.1 — Speaker switch detection

When `event::UserIdentified` reports a different user from
the previous turn, the FSM treats it as a fresh session for
context purposes (PromptBuilder includes the prior user's
turns as third-person context, not first-person).

### T12.3.1.2 — Interleaved STT

Mid-utterance speaker change is harder. For v1, ignore
within-utterance switches (the speaker who started owns the
utterance). v2 punts to a separate dogfood pass.

### T12.3.1.3 — FSM extension

Add a `SpeakerContext` field to FsmSnapshot. Transitions
update on `UserIdentified`. Manager rebuilds prompt context
on speaker change.

### T12.3.1.4 — Acceptance

Two enrolled users alternate ~5 turns in a conversation.
Each user's turn is correctly attributed; facts each
mentions go to the right user's memory; the LLM addresses
each by their enrolled name.
