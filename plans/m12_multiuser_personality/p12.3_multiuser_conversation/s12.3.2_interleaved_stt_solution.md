# S12.3.2 — Interleaved STT solution

**Phase:** P12.3 (multi-user conversation)
**Estimate:** 1 week
**Status:** stage placeholder.
**Shape:** investigation + impl, opens question Q24.

## Context

`ideas.md` line 107: "multi-user conversation -> interleaved
STT - solution???".

S12.3.1 says "ignore within-utterance switches for v1". This
stage is the v2 — what if two users *do* alternate inside one
utterance window? Two cases:

1. **Sequential** — user A finishes, user B starts, before
   the endpointer fires.
2. **Overlapping** — both speak simultaneously.

Case 2 is hard (diarization + source separation); case 1 is
the realistic one and the focus.

## Tasks

### T12.3.2.1 — Solution shapes (Q24)

Three options:
- **A)** Single STT stream + speaker-switch detection on
  the partial transcript stream — when voice ID changes,
  force-end the current utterance and start a new one.
  Simple, latency-cheap, may misattribute the boundary
  word.
- **B)** Per-speaker streaming sessions — when voice ID
  changes mid-utterance, fork a second STT session at the
  switch point. Two open Speaches sessions; coordinate
  to preserve word-order on output. Complex, robust.
- **C)** Diarization + post-segment — record the full
  utterance, run diarization (pyannote / sherpa-onnx
  diarization), produce segmented transcripts. Highest
  latency.

### T12.3.2.2 — Prototype the leading candidate

Pick A as default; build a diarized fixture set; measure
attribution accuracy + latency cost.

### T12.3.2.3 — Resolve Q24

Land the decision in
`m12_multiuser_personality/open_questions.md` Q24 with
measured numbers.

### T12.3.2.4 — Implementation

Whatever wins. A is the smallest impl; the rest scale up.
