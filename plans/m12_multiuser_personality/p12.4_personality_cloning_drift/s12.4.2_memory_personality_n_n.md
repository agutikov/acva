# S12.4.2 — Memory ↔ personality n-n optional relation

**Phase:** P12.4 (personality cloning + drift)
**Estimate:** 4-5 days
**Status:** stage placeholder.
**Shape:** schema + impl, opens question Q25.

## Context

`ideas.md` line 10: "memory-personality relations -
everything is optional: n-n relation".

Today: memory (sessions / turns / facts / summaries) is
implicitly per-session; personality is loaded as a config
overlay. There's no schema link between "this fact was
learned by personality X" or "this fact is shared across
personalities Y and Z".

The user wants a flexible cardinality: a fact can belong
to none, one, or multiple personalities. Personality A
might want only its own learned facts in context;
personality B might want shared general facts plus its
own; a "household" mode might want all facts.

## Tasks

### T12.4.2.1 — Q25 — cardinality choice

Three options:
- **A)** No link — personality reads all facts (status
  quo). Simple; loses personality-specific privacy.
- **B)** Strict per-personality scope — facts tagged with
  one personality, no cross-read. Strong isolation; loses
  shared knowledge.
- **C)** N-N optional — facts have an explicit
  `personality_set` (NULL = universal); personality
  context query honours the set. Most flexible.

Decision lands in `m12_multiuser_personality/open_questions.md`
Q25.

### T12.4.2.2 — Schema (assuming C)

New table `fact_personality(fact_id, personality_name)`.
Many-to-many. Fact rows without entries are universal.

### T12.4.2.3 — Tagging policy

When a fact is learned, which personalities does it
attach to? Default: the active personality. Operator can
override via tool call or admin command.

### T12.4.2.4 — Retrieval

PromptBuilder queries: facts where the row's
`personality_set` is NULL OR contains the active
personality. Indexed on `(personality_name, fact_id)`.

### T12.4.2.5 — Migration

Existing facts have no entries → universal. Operators can
re-scope after personality migration via a CLI tool.
