# S12.2.2 — Person state from memory to prompt

**Phase:** P12.2 (per-user memory + attribution)
**Estimate:** 4-5 days
**Status:** stage placeholder.
**Shape:** impl.

## Context

`ideas.md` line 125: "person state from memory to prompt".

Distinct from M11.P11.3 fact retrieval (which answers "what
do I know about X"): this stage is about *current state* of
the person — mood, presence, ongoing project, last topic —
fed forward into the system prompt every turn so the
assistant has continuity.

Imagine: the user opens the day saying "I'm tired today,
been pulling all-nighters". 30 turns later they ask "what's
up?". The assistant should remember the user is tired,
even though "tired" is not a fact retrieved by similarity
search.

## Tasks

### T12.2.2.1 — Person-state schema

New SQLite table `person_state(user_id, key, value,
updated_at, source_turn_id)`. Keys: `mood`, `current_topic`,
`current_project`, `last_seen_at`, …

### T12.2.2.2 — State extractor

LLM-side or rule-based: at end of turn, ask the LLM "did
this turn update the user's state? if so emit
`state_update(key, value)` actions". Reuses S9.4.3 GBNF
infrastructure.

### T12.2.2.3 — PromptBuilder injection

PromptBuilder reads active user's `person_state` rows and
injects a "Current state:" block in the system prompt.
Capped at N rows by recency.

### T12.2.2.4 — Decay policy

`mood` should decay (yesterday's tiredness shouldn't bias
today's conversation). Policy: timestamped staleness +
explicit decay rules per key.

### T12.2.2.5 — Acceptance

Operator says "I'm tired" → 10 turns later asks "remind me
why I'm not focused" → assistant references the earlier
"tired" without a similarity-search hit (the data flowed
via state, not facts).

## Resolves / depends

- Depends on M12.P12.1 voice ID for `user_id`.
- Pairs with M11.P11.3 fact retrieval — different access
  pattern, different recall semantics.
