# S9.1.3 — Smoke + acceptance

**Phase:** P9.1 (streaming partials source)  
**Estimate:** 2-3 days  
**Depends on:** S9.1.2 partials flowing on the bus.

## Context

Verification stage — closes the phase by demonstrating partials
are real, reasonably timely, and ready for P9.2 + P9.3 to
consume. Also adds the `acva demo partials` smoke command for
ongoing visibility.

## Acceptance

- New `acva demo partials` subcommand: 30 s mic capture, prints
  every `PartialTranscript` and `FinalTranscript` with timestamps.
  Exit summary: count of partials per turn, mean inter-partial
  interval, time from `SpeechStarted` to first partial.
- Acceptance numbers from the dev workstation:
  - First partial arrives within ≤ 500 ms after `SpeechStarted`
    on a 5+ word utterance (faster-whisper turbo on RTX 4060).
  - At least one partial per ~300 ms during sustained speech.
  - `FinalTranscript` text matches the last partial's text
    within edit distance ≤ 3 in 90% of fixtures (small
    revisions are normal).
- Live integration test: same as T9.1.2.4 but run from
  `./run_integration_tests.sh`.

### Task T9.1.3.2 — Latency budget verification

Write `scripts/partials-latency.sh` (one-off, lives in scripts/
not built into the binary): runs `acva demo partials` against
3 fixtures (short / medium / long utterances), parses output,
prints a table of first-partial-latency + inter-partial-mean +
final-vs-last-partial edit distance.

If first-partial latency > 800 ms or final-vs-last edit
distance > 5, that's a regression — file as a P9.1 sub-task
or escalate.
