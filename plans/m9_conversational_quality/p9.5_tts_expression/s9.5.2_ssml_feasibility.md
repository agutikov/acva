# S9.5.2 — SSML feasibility

**Phase:** P9.5 (TTS expression)
**Estimate:** 3-5 days
**Status:** stage placeholder.
**Shape:** investigation only.

## Context

`ideas.md` line 67: "SSML".

SSML (Speech Synthesis Markup Language) is the standard prosody
language. Cloud TTS (Google, Polly) supports the full grammar;
open TTS engines support partial subsets at best.

## Tasks

### T9.5.2.1 — Per-engine support matrix

For each engine acva ships against today and the realistic
candidates:

- Speaches (proxies Kokoro / Piper / OpenAI-spec backends)
- Piper direct
- Kokoro direct
- Coqui XTTS

document which SSML tags work, which silently no-op, which
break.

### T9.5.2.2 — Translation layer feasibility

If no engine supports a given tag (e.g. `<emphasis>`), can
we translate at the orchestrator? `<emphasis>` to a faster
+ louder TTS request slice with PauseClient bridging? Often
yes for `<break>` (just inject a silence chunk); rarely for
prosody tags.

### T9.5.2.3 — Decision write-up

Resolve in this stage:
- "ship full SSML translation layer" — large project.
- "ship `<break>` and `<sub>` as orchestrator-side rewrites"
  — small, useful.
- "skip — operators use prosody control (S9.5.5) instead" —
  defer.

Result lands in the stage as a closure note (no global
question; the decision is local to the survey output).
