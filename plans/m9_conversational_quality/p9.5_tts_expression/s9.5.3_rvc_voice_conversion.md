# S9.5.3 — RVC (Retrieval-based Voice Conversion)

**Phase:** P9.5 (TTS expression)
**Estimate:** 1-2 weeks (includes runtime build-out)
**Status:** stage placeholder.
**Shape:** investigation + gated impl.

## Context

`ideas.md` lines 65-71 (sound filters / artistic / sarcastic)
and explicit follow-up from the planning conversation.

RVC takes any TTS audio and conforms it to a target voice
embedding learned from a few minutes of reference audio.
Used in production by (e.g.) AI vtubers; common upstream
pipelines exist as Python wrappers around RVC ONNX
checkpoints.

GPU cost: ~300-500 MB VRAM idle, peaks under 1 GB. Tight on
the 8 GB budget alongside llama + Speaches; needs the model-
controller (M8A) to time-share.

## Tasks

### T9.5.3.1 — Upstream survey

Catalog viable RVC runtimes that work as a service (HTTP /
WebSocket) the orchestrator can talk to:

- `RVC-Project` original gradio (subprocess, slow startup).
- `Mangio-RVC-Fork` performance fork.
- `RVC-WebUI-Lite` minimal API.

Pick one to prototype against.

### T9.5.3.2 — Pipeline wiring

`audio::RvcSink` between TtsBridge resample and PlaybackQueue,
optional behind `cfg.tts.rvc.{enabled, model_path,
target_voice}`. When the RVC service is unhealthy, falls back
to bypass.

### T9.5.3.3 — VRAM coordination

RVC inference shares the GPU with llama + Speaches. Needs:
- model-controller integration (add `rvc` capability slot).
- Eviction policy: if RVC + STT both want the GPU during a
  turn, RVC waits.

### T9.5.3.4 — Decision

After T9.5.3.1-3 ship a prototype, decide:
- "ship as opt-in feature behind gate" — runs on operator-
  trained checkpoint only.
- "defer to post-MVP" — write up the prototype and what
  blocks general availability.

## Resolves / depends
