# S9.4.7 — XML / JSON output channel

**Phase:** P9.4 (LLM output hygiene)
**Estimate:** 4-6 days
**Status:** stage placeholder.
**Shape:** investigation + impl, opens question Q15.

## Context

`ideas.md` line 34: "GBNF and output formatting - XML, JSON".

The S9.4.3 GBNF action grammar gives us `think/wait/speak/
ignore`. Some agentic flows want structured *data* output
distinct from natural-language speech: a JSON payload to
display in the M10 Web UI, a small XML doc for a tool
manifest, a code block for execution.

This stage decides how data co-exists with action output —
and is the question M11.P11.2 (tool-call grammar) inherits.

## Tasks

### T9.4.7.1 — Survey existing options (Q15)

Three candidate shapes:

- **A)** Extend the action grammar with a `data(format,
  payload)` action. Single grammar, one parser.
- **B)** Separate streaming channel — chat-completions returns
  a `tool_calls` / `function_calls` array alongside content.
- **C)** Markdown code-fence convention — model emits
  ```` ```json ... ``` ```` blocks; parser recognises them.

Document trade-offs per axis: implementation cost, GBNF
expressiveness, M11.P11.2 reuse, model-side prompt clarity.

### T9.4.7.2 — Pick + implement

After T9.4.7.1, file the resolution in
`m9_conversational_quality/open_questions.md` Q15 and ship
the chosen path. Whatever wins must compose with S9.4.3
without breaking existing actions.

### T9.4.7.3 — Tests

Round-trip: Manager + parser handles `speak("...") +
data(json, "{...}")` interleaving without duplicating or
losing payload bytes. Malformed JSON inside the payload
fails cleanly.

## Resolves / depends

- **Resolves Q15** (XML/JSON channel choice).
- Forward-link: M11.P11.2 reuses the chosen mechanism for
  tool-call output. If A wins, M11.P11.2 just adds another
  action; B/C make M11.P11.2 a port instead of an
  extension.
