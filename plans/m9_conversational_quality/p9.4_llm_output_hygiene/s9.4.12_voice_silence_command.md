# S9.4.12 — Voice "SHUTUP" / silence command

**Phase:** P9.4 (LLM output hygiene)
**Estimate:** 4-5 days
**Status:** stage placeholder.
**Shape:** impl + opens question Q18.

## Context

`ideas.md` line 14: "SILENCE!!!! SHUTUP!!!". The user wants a
fast voice-driven way to make the assistant stop talking and
not respond. M8A already exposes `POST /mute` over HTTP, but
no voice equivalent.

Barge-in (M7) cancels the current turn but doesn't suppress
the next reply — it just lets the user start a new one.
"Shut up" is different: cancel + don't speak again until
explicit unmute.

## Tasks

### T9.4.12.1 — Pick gate (Q18)

Three options:
- **A)** Keyword spotter — porcupine-style or a tiny VAD-
  triggered fixed phrase classifier. Low latency.
- **B)** Real STT + LLM judge — whatever the user said is
  fed to the LLM with a system instruction "is this a
  request to be silent?". Robust to phrasing; slow.
- **C)** GBNF `silence` action — the LLM itself recognises
  the user's intent and calls `silence()`. Reuses S9.4.3
  grammar.

### T9.4.12.2 — Implementation

Whichever Q18 picks:
- A: Add tiny model under `models.wake_word` registry; new
  detector hooked alongside the existing wake-word
  pipeline.
- B: Manager-side intent check at FinalTranscript before
  routing to the LLM.
- C: GBNF action handler that toggles mute via the same
  path as `POST /mute`.

### T9.4.12.3 — Unmute path

Voice unmute is a separate question — keyword wake?
explicit "wake up"? web-UI unmute only? Document the
chosen unmute path in the same stage write-up.

### T9.4.12.4 — Tests

Acceptance: user mid-conversation says "shut up" → next
TTS chunk is suppressed within 200 ms; FSM enters Muted;
unmute via documented path within ≤ 1 s.
