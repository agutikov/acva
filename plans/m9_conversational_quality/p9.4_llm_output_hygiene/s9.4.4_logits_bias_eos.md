# S9.4.4 — Logits bias + EOS probability

**Phase:** P9.4 (LLM output hygiene)
**Estimate:** 3-4 days
**Status:** stage placeholder.
**Shape:** investigation + impl.

## Context

`ideas.md` lines 18-19: "logits bias / increase EOS probability".
Today llama-server runs default sampling. Operators have no
direct way to discourage filler tokens or encourage early
EOS to keep replies short.

## Tasks

### T9.4.4.1 — llama.cpp logit_bias survey

Document llama-server's `logit_bias` request field shape, what
the GBNF and chat-template paths apply on top, and where biasing
can backfire (token-id stability across model upgrades, BOS/EOS
id collisions for multi-language tokenizers).

### T9.4.4.2 — EOS-bias config field

Add `cfg.llm.eos_bias` (float, default 0.0) — applies to the
detected EOS token id at request time. Hot-reloadable.

### T9.4.4.3 — Filler-token deny list

Optional `cfg.llm.deny_tokens: [string]` — operator names
("Of course", "Certainly") that get translated to token-id
bias at request build time. The translation is brittle across
tokenizers; document the limitation.

### T9.4.4.4 — Acceptance

A/B on the soak driver prompt set: with `eos_bias=2.0` and
the M8C filler list, mean response length drops ≥ 15% and
filler-rate (per S9.4.1 metric) drops ≥ 30%.
