# S11.3.2 — Graph database investigation

**Phase:** P11.3 (Knowledge graph + vector store)
**Estimate:** 4-6 days
**Status:** stage placeholder.
**Shape:** investigation, opens question Q22.

## Context

`ideas.md` line 116: "different database, vector database,
graph database".

Q7 picks SQLite + sqlite-vec as the v1 default for the KG
because it inherits acva's existing memory-thread + path
plumbing. That's the right v1 choice but doesn't answer
whether richer graph stores (neo4j, kuzu, oxigraph) earn
their keep for richer queries.

## Tasks

### T11.3.2.1 — Workload survey

Catalog the queries M11.P11.3 + the KG query tool
(S11.4.1 T11.4.1.3) actually run. If they're 90% point
lookups + 10% 2-hop traversal, sqlite-vec is fine. If
deeper traversal (path-finding, transitive closure) shows
up, a graph store earns the slot.

### T11.3.2.2 — Candidate matrix

For each candidate (neo4j, kuzu, oxigraph, RDF stores),
document: deployment shape, RAM cost, query language,
license, packaging cost.

### T11.3.2.3 — Resolve Q22

Three options:
- **A)** Stay sqlite-vec only.
- **B)** Optional sidecar — operator runs neo4j alongside
  if their workload demands it; acva auto-detects.
- **C)** First-class swap — KG store is a config-selected
  backend.

Decision lands in `m11_tools_agent_platform/open_questions.md`
Q22.
