# S11.2.1 — `tool_call` grammar action

**Phase:** P11.2 (GBNF tool-call output)  
**Estimate:** 2-3 weeks  
**Status:** stage placeholder.  
**Depends on:** M9.P9.4 GBNF infrastructure.

## Likely tasks

### T11.2.1.1 — Extend the action grammar

Add `tool_call(name, args)` to M9.P9.4's grammar. `args` is a
JSON object validated against the tool's manifest input
schema.

### T11.2.1.2 — Parser extension

`src/llm/action_parser.cpp` emits
`event::ToolCall{name, args, turn_id}` for `tool_call`
actions; existing think/wait/speak/ignore handling unchanged.

### T11.2.1.3 — Manager routing

On `ToolCall`, Manager looks up the tool, validates
permissions, dispatches via the P11.1 framework. Result
flows back as `event::ToolResult` → Manager re-prompts the
LLM with the result included → continued generation.

### T11.2.1.4 — Multi-step / nested calls

A single LLM turn can chain `tool_call → think → tool_call →
speak`. Cap depth (`cfg.tools.max_call_depth`, default 5) to
prevent infinite loops. Cancellation cascades through the
chain on barge-in.

### T11.2.1.5 — Tests

Unit: tool_call parsing edge cases (escaped JSON, nested
arrays). Integration: chained call sequence, cancellation
mid-chain.
