# S10.4.1 — Config editor + memory CRUD

**Phase:** P10.4 (config editor + memory ops)  
**Estimate:** 2-3 weeks  
**Status:** stage placeholder.  
**Open question:** Q3 (config in DB vs YAML) — decided here.

## Likely tasks

### T10.4.1.1 — Decide on config storage (Q3)

YAML stays canonical OR move to SQLite. Evaluate against:
operator workflow ($EDITOR vs UI), atomicity, the existing
hot-reload story. Document the choice in
[`m10_web_ui/open_questions.md`](../open_questions.md) Q3.

### T10.4.1.2 — Schema-aware config form

Render every field in `cfg::Config` as a form input. Group
by section. Each field carries a "hot" or "restart-required"
chip from `src/config/reload.cpp` classification. Validate
in-browser using the OpenAPI schema; validate on submit
server-side.

### T10.4.1.3 — Apply path

`POST /api/v1/config` with the diff. Server runs the same
`diff_configs` + `apply_hot_fields` path SIGHUP uses;
returns 200 / 409 / 400 with the same shape. Restart-required
changes prompt the user to confirm + show the impact.

### T10.4.1.4 — Memory CRUD

UI surface for `acva memory <subcommand>`: list sessions,
drill into turns / facts / summaries, delete with confirm,
vacuum. Reuses the existing `cli/memory_cli` logic via the
new API.

### T10.4.1.5 — Privacy commands surfaced

Mute/unmute toggle (binds to
`POST /mute|/unmute`); "new session" button; "wipe session
N" / "wipe all" with double-confirm.
