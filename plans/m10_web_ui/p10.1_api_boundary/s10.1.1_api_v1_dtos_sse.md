# S10.1.1 — `/api/v1/*` DTOs + SSE event stream

**Phase:** P10.1 (API boundary)  
**Estimate:** 2-3 weeks  
**Status:** stage placeholder — to be expanded into 2-3 stages when the phase is picked up.

## Context

Foundational stage — establishes the HTTP contract between the
embedded Web UI and the orchestrator. Everything downstream
(P10.2–P10.5) depends on this surface.

Three surfaces:

- **`/api/v1/*` JSON DTOs** — stable, versioned, never expose
  internal C++ types. Read paths first (sessions, turns, config,
  status); write paths in P10.4.
- **SSE event stream** at `/api/v1/events` — fan-outs the
  internal `EventBus` to subscribers (FSM transitions,
  partial/final transcripts, LLM tokens, playback events).
- **OpenAPI spec** — generated or hand-rolled, lives at
  `docs/api/v1.yaml`. The frontend types come from this.

## Likely tasks (to be split into stages)

### T10.1.1.1 — DTO design pass

Enumerate every field the UI needs from each subsystem.
Write `src/http/api_v1/dtos.hpp` with one struct per
resource (`SessionDto`, `TurnDto`, `FactDto`, `StatusDto`,
`ConfigDto`, …). Keep them flat — no pointers, no
internal references.

### T10.1.1.2 — Read endpoints

`GET /api/v1/sessions`, `GET /api/v1/sessions/{id}/turns`,
`GET /api/v1/status` (extends current `/status`),
`GET /api/v1/config` (returns current config + reload
classification per field).

### T10.1.1.3 — SSE event stream

`GET /api/v1/events` — long-lived SSE connection. Multiplex
the subscribed event stream into JSON-encoded events.
Backpressure: bounded queue per subscriber; drop-oldest
on overflow with a `dropped_count` counter event.

### T10.1.1.4 — OpenAPI

Hand-roll `docs/api/v1.yaml` covering everything in
T10.1.1.1–3. Lint with `redocly` or similar.

### T10.1.1.5 — Tests

Unit tests for each DTO serialization round-trip. One
integration test that opens an SSE connection, drives a
synthetic FSM event sequence, asserts events arrive in
order.

## Out of scope (P10.4+)

Write paths (`POST /api/v1/config`, `PUT
/api/v1/sessions/{id}`) — separate stage with the
hot-vs-restart validation logic.
